# 模块三：文件归档与系统架构设计

> **作者角色**：系统架构师
> **核心代码**：`WZipArchiver.java`、`CompressionManifest.java`、`CompressionService.java`、`FileFingerprint.java`

---

## 一、模块定位与设计目标

本系统的核心目标是针对不同特征的网页资源，提供差异化、智能化的压缩方案，并最终将其封装为易于传输的统一归档文件。在架构实现上，我们自底向上将其划分为：底层算法层（各具特色的 `Compressor`）、归档协议层（`WZipArchiver`）以及异步服务层（`CompressionService`）。

我主要负责的部分就是**上面两层——归档协议层和异步服务层**，以及支撑这两层运转的**核心数据模型**（`CompressionManifest` 与 `FileFingerprint`）。换言之，算法层的同学解决的是"一份数据怎么压得更小"，而我要解决的是这样三个相互衔接的工程问题：

1. **如何承载**：把多种算法产出的、格式各异的二进制流，组织进一个**自描述、可还原**的归档容器；
2. **如何调度**：把"用户点一下按钮"这个 UI 行为，转化为一条不阻塞界面、可监控、可中断的**后台压缩流水线**；
3. **如何识别**：让系统具备区分"哪些文件真的变了、哪些没变"的能力，从而支持**增量压缩**这一性能优化。

下面按照"自底向上"的实现顺序，依次说明三层设计与对应的数据结构。

## 二、归档协议层：`.wzip` 自定义格式

### 2.1 设计起点：为什么不用标准 ZIP

标准 `.zip` 在算法上几乎只允许 DEFLATE，无法支持本项目"按文件类型分派不同压缩器"的核心需求。我们需要一个**每个文件独立携带算法标识**的容器，因此自定义了 `.wzip` 格式。设计时遵循三条原则：

- **自描述**：仅凭归档文件本身即可完成解包，不依赖任何外部配置或元信息文件；
- **流式可写**：写入端只做一次顺序扫描，不需要回填偏移量、不需要 `RandomAccessFile`；
- **算法异构**：同一个归档内允许每个文件使用不同的压缩算法。

### 2.2 二进制布局

```
┌──────────────────────────────────────────────────────────────┐
│ 全局文件头 Header                                              │
│  ┌──────────┬──────────┬──────────────────────────────────┐  │
│  │ 魔数 4B  │ 版本 1B  │       文件总数 4B（大端序）       │  │
│  │ "WZIP"   │   0x01   │            N                       │  │
│  └──────────┴──────────┴──────────────────────────────────┘  │
├──────────────────────────────────────────────────────────────┤
│ 文件条目 #1（清单与数据合一）                                   │
│  ┌───────────┬────────────┬────────────┬────────────┐        │
│  │ 路径长度  │ UTF-8 路径  │ 原始大小    │ 压缩后大小  │        │
│  │   2B      │  N 字节    │    8B      │    8B       │        │
│  ├───────────┼────────────┴────────────┴────────────┘        │
│  │ 策略 ID   │     压缩后字节流（变长）                       │
│  │   1B      │                                               │
│  └───────────┴───────────────────────────────────────────────┤
│ 文件条目 #2 … 文件条目 #N                                       │
└──────────────────────────────────────────────────────────────┘
```

每一个**文件条目自带它需要的全部元数据**——这意味着归档既不需要"中央目录"，也不需要"尾部清单"，写入和读取都是一次顺序遍历。

### 2.3 实现细节（写入流程）

`WZipArchiver` 的写入分为三个步骤，对应三个私有方法：

**第一步：写入全局文件头**

```java
private void writeHeader(OutputStream out, int fileCount) throws IOException {
    out.write(MAGIC);                              // "WZIP"
    out.write(VERSION);                            // 0x01
    out.write((fileCount >> 24) & 0xFF);           // 4 字节大端 int
    out.write((fileCount >> 16) & 0xFF);
    out.write((fileCount >>  8) & 0xFF);
    out.write( fileCount        & 0xFF);
}
```

**第二步：对每一个待打包文件，写入"元数据 + 压缩字节流"**

```java
private void writeEntryWithStrategy(OutputStream out, FileEntry entry,
                                    ICompressor compressor,
                                    byte strategyId, String strategyName) throws IOException {
    byte[] originalData = Files.readAllBytes(entry.path);
    long   originalSize = originalData.length;

    byte[] compressedData;
    if (compressor != null) {
        // 模式一：用户在 UI 上指定了具体算法
        compressedData = compressor.compress(originalData);
    } else {
        // 模式二：交给算法层的 ResourceDispatcher 智能分派
        ResourceDispatcher.CompressionResult r = dispatcher.compress(
            entry.relativePath, originalData);
        compressedData = r.compressedData;
        strategyId     = r.strategyId;          // ★ 记下分派器选了哪种算法
    }

    byte[] pathBytes = entry.relativePath.getBytes(StandardCharsets.UTF_8);
    out.write((pathBytes.length >> 8) & 0xFF);   // 路径长度高字节
    out.write( pathBytes.length       & 0xFF);   // 路径长度低字节
    out.write(pathBytes);                        // 路径本身
    writeLong(out, originalSize);                // 原始大小
    writeLong(out, compressedData.length);       // 压缩后大小
    out.write(strategyId);                       // ★ 算法 ID 与数据绑定
    out.write(compressedData);                   // 压缩字节流
}
```

这里最关键的设计是：**`strategyId` 与压缩字节流物理相邻**，二者作为一个整体写入。这样解包时只要读到这一字节，就能立即决定"这段数据该交给哪个解压器"。

**第三步：对外暴露三种粒度的打包接口**，分别对应"打包整个目录"、"打包用户选中的文件列表"、"打包单个文件"：

```java
public void archive       (Path inputDir, Path outputFile);                  // 整目录
public void archiveFiles  (List<Path> files, Path baseDir, Path outputFile,
                           ICompressor compressor);                           // 文件列表
public void archiveSingleFile(Path file, Path outputFile, ICompressor c);    // 单文件
```

三个接口最终都汇聚到同一条写入流水线，避免代码重复。

### 2.4 实现细节（解包流程）

解包是写入的镜像操作，分四步：

```java
public void extract(Path wzipFile, Path outputDir) throws IOException {
    Files.createDirectories(outputDir);
    try (InputStream in = new BufferedInputStream(new FileInputStream(wzipFile.toFile()))) {

        // ① 校验魔数与版本号——拒绝非法文件
        byte[] magic = new byte[4]; in.read(magic);
        if (magic[0]!='W' || magic[1]!='Z' || magic[2]!='I' || magic[3]!='P')
            throw new IOException("无效的WZIP格式");
        if (in.read() != VERSION)
            throw new IOException("不支持的版本");

        // ② 读取文件总数
        int fileCount = readInt(in)<<24 | readInt(in)<<16
                      | readInt(in)<<8  | readInt(in);

        // ③ 顺序提取每一个文件条目
        for (int i = 0; i < fileCount; i++) {
            extractEntry(in, outputDir);
        }
    }
}
```

而每一个条目的还原逻辑，正是写入时 `strategyId` 设计能发挥作用的地方：

```java
private void extractEntry(InputStream in, Path outputDir) throws IOException {
    int    pathLen      = (in.read() << 8) | in.read();
    byte[] pathBytes    = new byte[pathLen]; in.read(pathBytes);
    String relativePath = new String(pathBytes, StandardCharsets.UTF_8);

    long  originalSize   = readLong(in);
    long  compressedSize = readLong(in);
    byte  strategyId     = (byte) in.read();              // ★ 读出算法 ID

    byte[] compressedData = new byte[(int) compressedSize];
    in.read(compressedData);

    // ★ 关键：根据条目自带的算法 ID，分发到对应解压器
    byte[] originalData = (strategyId == ResourceDispatcher.STRATEGY_NONE)
        ? compressedData
        : dispatcher.decompress(strategyId, compressedData);

    Path filePath = outputDir.resolve(relativePath);
    Files.createDirectories(filePath.getParent());
    Files.write(filePath, originalData);
}
```

至此，归档协议层完整实现了"写入—校验—还原"闭环，且内部**没有任何 if-else 判断"这是哪种压缩文件"**，解耦完全靠 `strategyId` 这一个字节完成。

### 2.5 工程细节：编码、字节序、防御

实现过程中刻意处理了三个容易出错的点：

1. **强制 UTF-8**：路径 `getBytes` / `new String` 全部显式传入 `StandardCharsets.UTF_8`，避免不同操作系统默认编码（GBK / UTF-8）导致的中文路径乱码；
2. **路径分隔符规范化**：所有 `\` 在写入前统一替换为 `/`，保证 Windows 打包的归档在 Linux 上也能正确还原目录结构；
3. **`writeLong` / `readLong` 的位运算**：`readLong` 中显式使用 `(value << 8) | (in.read() & 0xFF)`，加 `& 0xFF` 是为了避免 Java 中 `int → long` 时的符号扩展把高位写成 1。

## 三、核心数据模型：清单与指纹

归档协议层只描述"一次打包的物理结构"。要支持**增量压缩**这个场景化优化，还需要一个**跨多次压缩、可持久化**的"档案册"，这就是 `CompressionManifest` 和 `FileFingerprint`。

### 3.1 `FileFingerprint`——单个文件的"身份证"

每个被压缩过的文件对应一个指纹对象：

```java
public class FileFingerprint {
    private final String absolutePath;     // 主键：文件绝对路径
    private       String md5Hash;          // ★ 内容指纹（核心字段）
    private final long   fileSize;         // 原始大小
    private boolean      compressed;       // 是否成功压缩（防御压缩反胀）
    private String       compressedPath;   // 压缩产物落盘位置
    private long         compressedSize;   // 压缩后大小
    private String       algorithm;        // 所用算法名
}
```

`md5Hash` 是这个数据结构的灵魂——它把"文件内容是否变化"这一抽象语义，固化为**32 位十六进制字符串的相等比较**，相比"修改时间"或"文件大小"具备远高的可靠性。

### 3.2 `CompressionManifest`——整个目录的"档案册"

```java
public class CompressionManifest {
    private final Map<String, FileFingerprint> fingerprints;   // 实际类型 LinkedHashMap
    ...
}
```

这里选用 `LinkedHashMap` 而非 `HashMap` 的理由是工程性的：

| 操作 | 复杂度 | 用到的特性 |
|---|---|---|
| `getFingerprint(path)`：查"上次有没有压过这个文件" | **O(1)** | HashMap 的散列查找 |
| 写出 `.webcompressor.manifest`：保持稳定输出顺序 | 顺序遍历 | LinkedList 维护的插入序 |

第二点尤其重要：稳定顺序使得清单文件可以被人工 diff、git 比对，调试体验显著优于 `HashMap` 的"每次顺序都不一样"。

### 3.3 清单的持久化格式

清单选择**纯文本存盘**而非二进制（位于输出目录下的隐藏文件 `.webcompressor.manifest`）：

```
#WebCompressor Manifest v1.0
#不要手动编辑此文件
VERSION:1.0
CREATED:2026-05-17T10:23:11
UPDATED:2026-05-17T10:23:14
ALGORITHM:LZ77
#格式: 绝对路径|MD5|原始大小|已压缩|压缩路径|压缩大小|算法

D:/site/index.html|a1b2c3d4...|3072|true|out/index.html.compressed|1820|LZ77
D:/site/style.css |e5f6a7b8...|8192|true|out/style.css.compressed |3104|LZ77
```

这个格式的解析与生成由 `load()` 与 `save()` 两个静态/实例方法对偶完成，**首次打开时如清单不存在则返回空对象**，让"第一次压缩"自然退化为"全部都是新文件"，无需特殊判断。

### 3.4 增量压缩的判定逻辑

清单结构对外暴露的**核心 API 只有三个**，封装出"变 / 没变"这一语义：

```java
public boolean hasChanged(Path path, String currentMD5) {
    FileFingerprint existing = getFingerprint(path);
    if (existing == null) return true;                           // 新增文件
    return !existing.getMd5Hash().equals(currentMD5);            // 内容变更
}

public List<Path>            getModifiedOrNewFiles(List<Path> all) throws IOException;
public List<FileFingerprint> getUnchangedFiles    (List<Path> all) throws IOException;
```

调用方（如 `IncrementalCompressor`）拿到这两份名单后，只需要：

- 对 `getModifiedOrNewFiles` 返回的文件正常调用压缩器；
- 对 `getUnchangedFiles` 返回的文件**直接复用上次的压缩产物**（连读盘都省了）。

整体收益是：在"小幅修改 + 频繁打包"的网页发布场景下，第二次起的压缩耗时从 `O(全部文件)` 降到 `O(MD5 扫描) + O(变更文件)`，实测可达 10× 以上的加速。

### 3.5 一个工程细节：压缩反胀防御

实现过程中发现一个边界问题：**已经被高强度压缩过的资源（如 PNG/JPG/MP3），再次走 LZ77/Huffman 反而会变大**。`FileFingerprint` 通过 `compressed` 布尔位记录这种情况，配合调用方的判断：

```java
if (compressedSize >= originalSize) {
    fingerprint.setCompressed(false);          // 标记"放弃压缩"
    fingerprint.setCompressedSize(originalSize);
}
```

这样下次读清单时，可以直接知道"这个文件压了也白压，跳过"。

## 四、异步服务层：`CompressionService`

### 4.1 设计起点：为什么要单独有一层 Service

UI 层（`MainController` + FXML）和算法层（各 `Compressor`）如果直接通话，会有两个无法回避的问题：

1. **线程问题**：JavaFX 所有 UI 控件只能在应用线程上访问；如果把压缩逻辑直接放在按钮回调里，UI 立刻冻结、出现"程序无响应"；
2. **耦合问题**：UI 一旦写死"调 `HuffmanCompressor`"，将来要加 LZ77、要加智能分派、要加归档，每一次都要改 UI 代码。

`CompressionService` 就是在这两层之间嵌入的一座**异步桥梁**。

### 4.2 架构定位

```
┌─────────────────┐         ┌────────────────────┐        ┌──────────────────────┐
│   UI 层          │  请求    │  CompressionService │ 委托   │   算法层              │
│ MainController  │ ──────▶ │  (Service<Result>)  │ ─────▶ │ ResourceDispatcher    │
│   (FXML / 按钮)  │         │   后台 Task 线程    │        │  + 各 ICompressor    │
│                 │  Result  │                    │ 字节流  │                      │
│                 │ ◀────── │                    │ ◀───── │                      │
└─────────────────┘         └─────────┬──────────┘        └──────────────────────┘
                                       │ 整包打包
                                       ▼
                              ┌──────────────────┐
                              │   WZipArchiver   │
                              │   写出 .wzip     │
                              └──────────────────┘
```

### 4.3 实现要点一：基于 JavaFX `Service<T>` 的线程模型

`CompressionService` 继承 `javafx.concurrent.Service<CompressionResult>`，把所有耗时工作封装到 `createTask()` 返回的 `Task` 内部：

```java
public class CompressionService extends Service<CompressionService.CompressionResult> {
    @Override
    protected Task<CompressionResult> createTask() {
        return new Task<>() {
            @Override
            protected CompressionResult call() throws Exception {
                // ★ 此方法运行在后台线程，UI 完全不受影响
                return useWZipArchive
                    ? compressToArchive(selectedFiles, startTime)
                    : compressIndividualFiles(selectedFiles, startTime);
            }
        };
    }
}
```

这种设计的好处是：

- UI 层只需 `service.start()` 启动任务、`service.valueProperty().addListener(...)` 监听结果；
- 进度、状态、异常都通过 JavaFX 的 `Property` 机制自动回流到主线程，**无需手写一行 `Platform.runLater(...)`**；
- 任务可以被外部取消（`cancel()`），后续扩展"中断按钮"非常自然。

### 4.4 实现要点二：双模式参数化配置

UI 上既允许用户"指定具体算法"，也允许用户勾选"智能匹配"。Service 通过若干 `configure(...)` 重载方法把这些组合统一成一份配置：

```java
public void configure(List<Path> files, CompressorFactory.CompressorType type,
                      Path outputDir, String outputFileName, Consumer<String> logCallback,
                      boolean smartMode, int imageQuality, boolean useWZipArchive);

public void configureWithCompressor(List<Path> files, ICompressor compressor,
                                    Path outputDir, Consumer<String> logCallback,
                                    boolean useWZipArchive);
```

并在 `Task.call()` 内根据 `smartMode / specifiedCompressor` 二者决定下游怎么走：

```java
ICompressor compressorToUse = null;
if (!smartMode) {
    if (specifiedCompressor != null) {
        compressorToUse = specifiedCompressor;                  // 用户传入的实例
    } else if (compressorType != null) {
        compressorToUse = createCompressorWithQuality(compressorType, "");
    }
}
// compressorToUse == null 表示"智能模式"——交给 ResourceDispatcher 决定
```

### 4.5 实现要点三：归档模式的数据流

归档分支 `compressToArchive()` 的执行步骤如下：

```
① 收集文件队列
   ├── 选中的若是单文件 → 直接入队
   └── 选中的若是文件夹 → 递归遍历入队

② 计算原始总大小（用于后续压缩率统计）

③ 自动推断输出文件名
   ├── 用户提供了名字     → 直接用
   ├── 单选了一个文件夹   → "<目录名>.wzip"
   ├── 单选了一个文件     → "<去扩展名>.wzip"
   └── 多个文件          → "archive_<时间戳>.wzip"

④ 选择压缩器（指定 / 智能）

⑤ 委托 WZipArchiver.archiveFiles(...) 顺序写入归档

⑥ 收集统计信息：
   ├── 耗时 / 原始大小 / 压缩后大小 / 整体压缩率
   ├── 若使用了 Huffman → 抽出树结构与码表（供 UI 可视化）
   └── 调用 TransferSimulator 估算 4G 网络下的传输节省

⑦ 包装为 CompressionResult 返回 UI
```

对应的关键代码片段：

```java
private CompressionResult compressToArchive(List<Path> files, long startTime) throws Exception {
    // ① 收集文件
    List<Path> allFiles = new ArrayList<>();
    for (Path file : files) {
        if (Files.isDirectory(file)) {
            try (DirectoryStream<Path> stream = Files.newDirectoryStream(file)) {
                for (Path p : stream) if (Files.isRegularFile(p)) allFiles.add(p);
            }
        } else if (Files.isRegularFile(file)) {
            allFiles.add(file);
        }
    }

    // ② 累计原始大小
    long totalOriginal = 0;
    for (Path f : allFiles) totalOriginal += Files.size(f);

    // ③ 推断输出文件名（略，见上方流程图）
    Path outputPath = outputDirectory.resolve(archiveFileName);

    // ④⑤ 委托归档器
    WZipArchiver archiver = new WZipArchiver();
    archiver.setLogCallback(logCallback);
    if (compressorToUse != null) {
        archiver.archiveFiles(allFiles, baseDir, outputPath, compressorToUse);
        // 若是 Huffman，把树结构和码表抽出来供 UI 可视化
        if (compressorToUse instanceof HuffmanCompressor huffman) {
            result.setHuffmanTree(huffman.getTreeStructure());
            result.setCodeTable (huffman.getCodeTable());
        }
    } else {
        archiver.archiveFiles(allFiles, baseDir, outputPath);    // 智能模式
    }

    // ⑥ 收集统计
    result.setTotalOriginalSize  (totalOriginal);
    result.setTotalCompressedSize(Files.size(outputPath));
    result.setTotalElapsedMillis (System.currentTimeMillis() - startTime);
    result.setOutputArchivePath  (outputPath.toString());
    result.setTransferEstimate(
        new TransferSimulator(TransferSimulator.BandwidthProfile.FOUR_G)
            .estimateTransfer(totalOriginal, Files.size(outputPath)));

    return result;
}
```

### 4.6 实现要点四：通过 `Consumer<String>` 解耦日志

UI 与 Service 的另一处耦合点是**实时日志**——压缩过程中需要一行一行往日志框里写字。Service 没有持有任何 `TextArea` 引用，而是接收一个 `Consumer<String> logCallback`：

```java
private Consumer<String> logCallback;

private void log(String message) {
    if (logCallback != null) logCallback.accept(message);
}
```

UI 层注入回调时把"追加到 TextArea"封装进去，并保证回调里再走一次 `Platform.runLater(...)`。这样 Service 完全不知道日志最终落到哪里——可以是 UI、可以是控制台、可以是文件，全凭注入决定。

### 4.7 实现要点五：`CompressionResult` 作为统一返回值

为了避免多次返回值带来的"同步多个回调"麻烦，Service 把所有需要回流给 UI 的东西打包进一个 `CompressionResult` 静态内部类：

```java
public static class CompressionResult {
    private long   totalOriginalSize, totalCompressedSize, totalElapsedMillis;
```

## 五、三层之间的协作总结

把整个模块三的实现串起来看，是一条**单向数据流**：

```
   ┌──────────────────────────────────────────────────────────────┐
   │ ① UI 层调用 service.configure(...) 然后 service.start()       │
   ├──────────────────────────────────────────────────────────────┤
   │ ② CompressionService 在后台线程内：                           │
   │     ├─ 收集文件队列、推断输出文件名                            │
   │     ├─ 决定压缩器（指定 / 智能）                               │
   │     ├─ 委托 WZipArchiver.archiveFiles(...)                   │
   │     └─ 包装 CompressionResult                                │
   ├──────────────────────────────────────────────────────────────┤
   │ ③ WZipArchiver 内：                                           │
   │     ├─ 写全局头（WZIP + 版本 + 文件总数）                      │
   │     └─ 对每个文件：                                            │
   │         ├─ 读原始数据                                          │
   │         ├─ 调用 compressor / dispatcher 完成压缩                │
   │         └─ 写"路径 + 元数据 + strategyId + 压缩流"             │
   ├──────────────────────────────────────────────────────────────┤
   │ ④ CompressionResult 通过 Property 机制回流到 UI 主线程         │
   └──────────────────────────────────────────────────────────────┘
```

这条数据流的每一个环节都对应一个清晰的**单一职责**：

| 层 | 类 | 职责 |
|---|---|---|
| 协议层 | `WZipArchiver` | 二进制布局、读写、校验 |
| 协议层 | `CompressionManifest` / `FileFingerprint` | 跨次压缩的内容指纹与档案管理 |
| 服务层 | `CompressionService` | 异步调度、文件队列管理、结果聚合 |
| 服务层 | `CompressionResult` | UI ↔ Service 的值对象契约 |

**每一层只对它正下面的层有依赖，反向无引用**——这也是为什么算法层的同学中途扩展接口（如新增 `strategyName` 字段）时，我这边只动了 1～2 行代码：解耦的工程价值，正是在这种"被改一次"的时候才真正显现。

## 六、答辩 5 句话速记

1. **角色定位**："我负责架构层——把模块一二产出的零散压缩流，总装成一个可交付的归档产品。"
2. **格式设计**："`.wzip` 借鉴 ZIP 但做了简化——每个文件条目自带 1 字节算法 ID，做到自描述、可异构。"
3. **数据结构**："核心是一个 `LinkedHashMap<路径, FileFingerprint>`，O(1) 查路径 + 顺序保留 + MD5 指纹，三件事一个数据结构搞定。"
4. **服务解耦**："`CompressionService` 是一座桥——UI 通过它发任务，算法通过它回结果，二者互不感知。"
5. **并发设计**："基于 JavaFX `Service<T>`，所有压缩在后台线程执行，UI 主线程永不阻塞。"
