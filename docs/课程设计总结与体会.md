# 课程设计总结与体会

> **课题名称**：WebCompressor —— 面向网页资源的多算法压缩与归档系统
> **个人模块**：模块三 文件归档与系统架构设计
> **核心代码**：`WZipArchiver.java` / `CompressionManifest.java` / `CompressionService.java`

---

## 一、课程设计过程的整体收获

这次大作业不是单纯"实现某个数据结构"，而是要把**树（Huffman）、哈希表（LZ77 滑动窗口、Manifest 索引）、字典树（LZW）、链表/数组（清单结构）**等多种结构融合到一个真实可运行的系统里。我的体会主要有三点：

### 1. 数据结构不是孤立的题目，而是工程组件

以前做题时遇到 Huffman 树就是"建树 → 编码 → 解码"三步走，但在做归档器时我才意识到——树/编码表怎么序列化、解压时怎么从字节流里把树原样还原回来、以及树所占的元数据空间是否会反过来吃掉压缩收益——这些"工程上的小事"才是真正决定数据结构能不能落地的关键。我在写归档元数据时还专门加了一行日志：

```
[智能匹配诊断] 文件: index.html | 扩展名: .html | 策略ID: 1 | 算法: LZ77
原始压缩率:42.3% | 净压缩率(含元数据27B):41.2%
```

把"原始压缩率"和"净压缩率"分开打印出来后，我才直观看到：**对小文件来说，元数据本身就是压缩损耗的主要来源**——这是只做单算法实验时永远不会注意到的细节。

### 2. 设计先于编码，比想象中更重要

第一版 `.wzip` 我图省事，用了"开头集中存清单 + 末尾存数据"的格式。结果发现写入时必须先扫一遍所有文件、计算每段数据的偏移量，才能回头去写清单——这等于多了一次完整的 IO 遍历，而且代码里到处是 `RandomAccessFile.seek()`，巨难调。

后来重新设计成**"流式条目（每个条目自带元数据）"**之后，写入和读取都退化成一个简单的 for 循环，代码量减少了一半，bug 也几乎消失。这件事让我深刻理解了一句老话：**"花一个小时画格式图，能省下三天调试时间。"**

### 3. 解耦的价值要靠"被改一次"才会真正体会到

模块二的同学中途调整了 `ResourceDispatcher` 的 API（多加了一个返回字段 `strategyName`）。因为我在 `CompressionService` 里把调度器当作接口依赖、而不是直接 new 具体压缩器，所以我只在两处加了字段读取，其他地方一动没动。如果当初是把所有压缩逻辑塞进 UI 控制器里，这一次改动至少要改 5 个文件。这让我对"面向接口编程"和"分层"不再只是概念上的理解。

## 二、遇到的问题 & 解决问题的思考过程

### 问题 1：解包后中文路径乱码

**现象**：把含 `index.html`、`样式表.css` 的目录打包，再解包后发现中文文件名变成了一串 `??` 或乱码。

**排查思路**：
1. 先怀疑读写过程中编码不一致，加日志打印写入和读取时的字节序列；
2. 发现写入是 `String.getBytes()`（默认平台编码 GBK），读取是 `new String(bytes)`（同样默认平台编码）——本机能跑，但不同平台必崩；
3. 改成统一 `StandardCharsets.UTF_8` 后问题消失。

**反思**：默认编码就是个隐形地雷，**只要涉及字节 ↔ 字符串的转换，永远显式声明 UTF-8**。这条经验我现在写任何 Java 项目都会遵守。

### 问题 2：`writeLong` 写入后再读出来变成了负数

**现象**：原始大小 5 GB 的目录，归档元数据里读出来变成 -3xxx... 的负数。

**排查思路**：
1. 第一反应是 `long` 不够用——但 Java 的 `long` 是 64 位、远远够用；
2. 单步调试发现 `readLong` 里写成了 `(int) in.read() << 56`，**移位运算时被强转回了 int**，高位全部丢失；
3. 改成 `((long) in.read() & 0xFF) << 56` 后正常。

**反思**：Java 的 `<<` 运算结果由**左操作数类型**决定——这是一个"看一眼觉得没问题、运行起来直接错"的经典坑点。后来我把所有移位前都习惯性加一个显式 `(long)`，并且 `& 0xFF` 防止符号扩展。

### 问题 3：UI 在压缩大文件时整个卡死

**现象**：选了一个 200 MB 的目录开始压缩，进度条不动、按钮不响应、最后还触发了 Windows 的"程序无响应"提示。

**排查思路**：
1. 检查代码发现压缩逻辑直接写在 `Button.setOnAction()` 里——也就是说**压缩跑在 JavaFX 应用线程上**，所有 UI 事件都被堵住了；
2. 改造成 `extends Service<CompressionResult>`，把压缩逻辑搬进 `Task.call()` 里；
3. UI 线程只负责：监听 `valueProperty()` 拿结果，监听 `progressProperty()` 更新进度条。

**反思**：以前学多线程只停留在 `new Thread().start()`，这次第一次真正用上 **生产者-消费者** 思维——UI 是消费者、Task 是生产者，二者通过 JavaFX 的 Property 机制（一种观察者模式）解耦。这之后我看任何 GUI 框架都会先去找它的"工作线程模型"长什么样。

### 问题 4：归档大目录时偶发 `OutOfMemoryError`

**现象**：测试一个 1 GB 的纯文本目录时，JVM 直接 OOM。

**排查思路**：
1. 一开始写的是 `Files.readAllBytes()`——一次性把整个文件读进内存；
2. 200 MB 的文件还能扛住，500 MB 就开始 GC 抖，1 GB 直接 OOM；
3. 改造成"读一个文件 → 压缩 → 写出 → 释放引用"的流式处理后稳定下来。

**反思**：这是我第一次直观感受到**算法复杂度（O(n)）和内存占用是两件事**——再快的算法，如果一次性把所有数据塞内存，也照样翻车。

## 三、对程序调试能力的思考

走完整个项目，我对"调试"这件事的理解发生了三个明显变化：

### 1. 从"加 println" 到 "结构化日志"

最初我满代码塞 `System.out.println("here")`、`System.out.println("x = " + x)`，跑一次满屏滚动根本看不清。后来我在 `WZipArchiver` 里规范了日志格式：

```
[智能匹配诊断] 文件: ... | 扩展名: ... | 策略ID: ... | 原始: ... | 压缩后: ...
```

每条日志都带着**统一的字段标签**，不仅好读，还能直接 grep。这种"日志即数据"的思路，让我后期排查问题时**根本不需要再启动调试器**——日志就够了。

### 2. 从"单步调试"到"二分定位"

调试 `writeLong` 那个 long 截断 bug 时，最早我是从头到尾单步走，看了二十多步才发现问题。后来再遇到类似问题，我会先做一件事：**在出错的输入和正常的输入之间二分**——比如先用 1 KB 文件试，再用 1 MB，再用 100 MB，定位到"什么规模开始出问题"，问题域立刻缩小一个数量级。

### 3. 从"修 bug" 到"造可复现的最小用例"

`extractEntry` 解出乱码那次，我没有直接去原 200 MB 目录里调，而是先写了一个 5 行的最小测试：建一个只有"中文.txt"的目录、打包、解包、对比。最小用例一旦能稳定复现，bug 基本就死了一半。这种思路后来也帮我在 GitHub 上写过几个能让别人秒看懂的 issue。

### 4. 学会"信任异常栈，但不全信"

`OutOfMemoryError` 那次，栈顶指向的是 `Files.readAllBytes` 里的一个 native 调用——但**根因不在那一行**，而在我的设计选择（一次性读全部）。栈只能告诉你"哪里崩了"，不能告诉你"为什么会走到那里"。这之后我看异常栈会习惯性往上**看三层调用方**。

## 四、课程设计实现过程中的体会

**1. 真实工程里的"复杂度"不是 Big-O，而是耦合度。**
Huffman 编码再优雅，如果它的输出格式没人能解出来，那它就是个废物；反过来，再朴素的算法只要接口干净，都能在团队里活得很好。这次我做架构师的最大体会就是——**好的接口让团队跑得起来，好的算法让单测跑得起来，二者缺一不可**。

**2. 写文档的过程，本身就是查 bug 的过程。**
我画 `.wzip` 格式的 ASCII 图时，一边画一边发现："咦，路径长度我用了 2 字节（最大 65535），但 UTF-8 中文一个字 3 字节，岂不是只能存 2 万字符的路径？" 这是写代码时我从来没意识到的边界——画图逼我把所有字段的取值范围都过一遍。

**3. 代码的可读性比聪明更重要。**
我曾经把魔数比较写成 `Arrays.equals(magic, MAGIC)`，看起来很优雅；但后来改成 `magic[0]!='W' || magic[1]!='Z' || magic[2]!='I' || magic[3]!='P'` 之后，**别人读代码时一眼就知道这是在校验 "WZIP" 这四个字符**。这种小细节让团队协作时的沟通成本下降了一个台阶。

**4. 软件工程是"管理不确定性"的艺术。**
模块二的算法实现一直在迭代——这周加了 LZW，下周加了 WebDict，再下周改了 Brotli 接口。如果我每次都跟着改，整个项目永远收敛不了。但因为我把 `ICompressor` 接口和"策略 ID"两层抽象抓得比较紧，**只要他们不破坏接口契约，我就能"无知地"继续往前走**。这是这门课让我最受启发的地方——**抽象的本质，是把别人的不确定性挡在自己外面**。

**5. 完成一个"看得见摸得着"的产品，比刷一百道 OJ 更让人有成就感。**
最后看着自己设计的 `.wzip` 文件能被双击 → 解出原网页 → 浏览器正常打开的那一瞬间，比平时 AC 一道困难题的快感要强很多——因为它证明的不只是某个算法对了，而是**整套系统从设计、实现、调试、到交付，每一环我都经手过、都跑通了**。这种端到端的体验，是这门课最珍贵的礼物。
